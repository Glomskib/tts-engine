1. EXECUTIVE SUMMARY
================================================================================

FlashFlow AI is a full-stack SaaS platform for TikTok content creators and
e-commerce brands. It automates the entire lifecycle of short-form video content:

  Script Generation → Avatar Video Rendering → Quality Scoring →
  Auto-Posting → Performance Tracking → Revenue Intelligence

The platform is built as a Next.js 16 monolith deployed on Vercel, backed by
Supabase (PostgreSQL + Auth + Storage), with 20 cron jobs orchestrating
background automation. It integrates with 12+ external services including
HeyGen (avatar videos), Runway (AI video), Anthropic Claude (AI brain),
Stripe (billing), TikTok (3 separate integration types), Telegram (alerts),
and more.

Key numbers:
  - 569 API routes
  - 273 lib modules
  - 196 database migrations (100+ tables)
  - 20 Vercel cron jobs
  - 67+ React components
  - 8 subscription tiers
  - 12+ external service integrations

4. VIDEO GENERATION PIPELINE
================================================================================

The video pipeline is FlashFlow's core product. It takes a product/hook idea
and produces a fully rendered, ready-to-post TikTok video.

── Pipeline Flow ───────────────────────────────────────────────────────────────

  1. SCRIPT GENERATION
     └─ lib/unified-script-generator.ts (25.5 KB)
     └─ Uses Claude Sonnet
     └─ Input: product ID, content type, persona, sales approach, hook
     └─ Output: hook, setup, body, CTA, on-screen text, captions, hashtags,
        filming notes
     └─ Draws from winner patterns + audience persona + creator style
     └─ Intelligent variety biased toward proven patterns

  2. SCRIPT SCORING
     └─ AI evaluates script quality (0-10 scale)
     └─ Auto-retry up to 3x if score < 7
     └─ Scorer feedback fed into regeneration prompts

  3. TEXT-TO-SPEECH
     └─ ElevenLabs voice synthesis
     └─ Persona-to-voice mapping (lib/heygen-personas.ts)

  4. AVATAR VIDEO (HeyGen)
     └─ lib/heygen.ts
     └─ Upload audio → POST /v2/video/generate → poll status → download MP4
     └─ 7 personas with specific avatar + voice combos:
        • Skeptic → Aditya + Brian (deep)
        • Educator → Aditya + Eric (smooth)
        • Hype Man → Aditya + Liam (energetic)
        • Sober Curious → Abigail + Jessica (warm)
        • Chronic Warrior → Abigail + Sarah (reassuring)
        • Honest Reviewer → Abigail + Matilda (professional)
        • Relatable Friend → Abigail + Laura (casual)
     └─ Output: 9:16 vertical (TikTok/Reels/Shorts)

  5. AI VIDEO (Runway) — optional
     └─ lib/runway.ts
     └─ Text-to-video and image-to-video
     └─ Models: gen3a_turbo, gen4.5, veo3, veo3.1, veo3.1_fast
     └─ 9:16 vertical default

  6. VIDEO COMPOSITION (Shotstack)
     └─ lib/shotstack.ts, shotstack-compose-v2.ts
     └─ Timeline-based composition of avatar + B-roll + text overlays
     └─ Output: MP4, SD, 9:16, 30fps

  7. QUALITY GATE
     └─ Automated quality check before publishing
     └─ Videos move through states: RENDERING → READY_FOR_REVIEW →
        APPROVED → READY_TO_POST → POSTED

── Auto-Generate Orchestrator ──────────────────────────────────────────────────

  Endpoint: POST /api/pipeline/auto-generate
  Auth: Admin session or CRON_SECRET Bearer token

  6-step orchestration:
    Step 1: Generate UGC script (unified script generator)
    Step 2: Score script (retry ≤3x if <7)
    Step 3: Generate HeyGen avatar video (async)
    Step 4: Generate B-roll via Runway (TODO)
    Step 5: Shotstack compose (async cron)
    Step 6: Quality gate (async cron)

  Features:
    - Correlation ID tracking for debugging
    - Telegram notifications at each stage
    - Logs to video_events table

── Check-Renders Cron ──────────────────────────────────────────────────────────

  Schedule: Every 2 minutes
  Endpoint: /api/cron/check-renders

  3 phases:
    Phase 1: Poll HeyGen for rendering status
    Phase 2: Poll Shotstack for composition status
    Phase 3: Move completed videos to next lifecycle state

── Auto-Post Cron ──────────────────────────────────────────────────────────────

  Schedule: Every 15 minutes
  Endpoint: /api/cron/auto-post

  Picks READY_TO_POST videos and publishes to TikTok via Content Posting API.
  Respects posting schedule and rate limits.

── Winner Detection ────────────────────────────────────────────────────────────

  Analyzes posted video performance metrics.
  Identifies "winner" scripts/hooks for future generation patterns.
  Winners Bank: /admin/winners — manages winning templates.

5. REVENUE INTELLIGENCE SYSTEM
================================================================================

Automated TikTok comment monitoring → AI classification → reply drafting →
urgency alerting. This is a full autonomous agent for comment-based revenue
capture.

── Pipeline ────────────────────────────────────────────────────────────────────

  1. COMMENT INGESTION
     └─ lib/revenue-intelligence/comment-ingestion-service.ts
     └─ Playwright scraper visits TikTok creator profiles
     └─ Scrapes recent video comments (configurable max per video/account)
     └─ Deduplicates via platform_comment_id
     └─ Stores in ri_comments (with is_processed flag)

  2. COMMENT CLASSIFICATION (Claude Haiku)
     └─ lib/revenue-intelligence/comment-classification-service.ts
     └─ Categories:
        • buying_intent  — wants to buy, asks price/link
        • objection      — skepticism, concerns
        • shipping       — delivery questions
        • support        — product issues, returns
        • praise         — positive feedback
        • troll          — spam, hate, irrelevant
        • general        — everything else
     └─ Scores: lead_score (0-100), urgency_score (0-100), confidence (0-100)
     └─ Batch: max 20 comments per API call
     └─ Stored in ri_comment_analysis

  3. REPLY DRAFT GENERATION (Claude Haiku)
     └─ lib/revenue-intelligence/reply-draft-service.ts
     └─ 3 drafts per comment:
        • neutral     — professional, non-pushy
        • friendly    — warm, casual, emoji-friendly
        • conversion  — sales-leaning, includes CTA
     └─ TikTok limit: 150 chars per reply
     └─ Stored in ri_reply_drafts

  4. AUTO-DRAFT (Gated)
     └─ lib/revenue-intelligence/auto-draft-service.ts
     └─ Selectively drafts high-scoring items first (lead >= threshold)
     └─ Runs before standard reply generation
     └─ Env-gated via RI_AUTO_DRAFT_ENABLED

  5. URGENCY SCORING
     └─ lib/revenue-intelligence/urgency-scoring-service.ts
     └─ Thresholds:
        • buying_intent + lead_score ≥ 70 → URGENT
        • objection + urgency_score ≥ 60 → URGENT
        • support + urgency_score ≥ 50 → URGENT
        • shipping + urgency_score ≥ 60 → URGENT
        • Any comment with ≥ 50 likes → URGENT (viral)
     └─ Flags in ri_comment_status

  6. ACTIONS QUEUE
     └─ lib/revenue-intelligence/actions-queue-service.ts
     └─ Enqueues items for human review
     └─ Priority scoring for inbox ordering

  7. ALERT POLICY → TELEGRAM DIGEST
     └─ lib/revenue-intelligence/alert-policy.ts (pure logic, zero deps)
     └─ Modes: off | urgent | batch | digest | all
     └─ SINGLE alert path: run-ingestion.ts → evaluateAlertPolicy() →
        sendDigestAlert() — no per-comment Telegram spam
     └─ Env config: RI_ALERT_MODE, RI_ALERT_BATCH_MIN, RI_ALERT_DIGEST_MINUTES

── Orchestration ───────────────────────────────────────────────────────────────

  Entry: scripts/revenue-intelligence/run-ingestion.ts
  CLI modes:
    pnpm ri:ingest              # Full live run
    pnpm ri:ingest:sim          # Simulation mode (mock data)
    pnpm ri:ingest:dry          # Dry run (no DB writes)

  Also: pnpm ri:ingest:daemon   # Long-running daemon (every 10-15 min)

  Safety: File-based run lock (lib/revenue-intelligence/run-lock.ts)
    - PID-aware, 10-minute TTL
    - Prevents daemon + manual overlap

── Revenue Inbox (UI) ──────────────────────────────────────────────────────────

  lib/revenue-intelligence/revenue-inbox-service.ts
  - Comment status: unread → reviewed → resolved
  - Filter by category, urgency, video
  - Reply draft selection and posting

── Database Tables ─────────────────────────────────────────────────────────────

  ri_creator_accounts    Active TikTok accounts to monitor
  ri_videos              Scraped videos
  ri_comments            Raw comments (ingested_at, is_processed)
  ri_comment_analysis    Classification + scoring results
  ri_reply_drafts        3 tone variants per comment
  ri_comment_status      Unread/reviewed/resolved tracking
  ri_actions_queue       Prioritized human review queue
  ri_run_state           Last ingestion state (incremental updates)
  ri_agent_logs          Structured activity logs

  honest_reviewer Calm, measured, tried everything
  educator        Confident, drops science
  storyteller     Narrative-driven, personal journey

  relatable_friend Casual, friend texting, low-key sell
  custom          User-defined persona

── Voice Tones ─────────────────────────────────────────────────────────────────

  authoritative   Expert confidence, facts-first
  empathetic      Understanding, validates feelings
  high_energy     Excited, fast-paced
  educational     Clear, informative
  raw_authentic   Unfiltered, stream of consciousness

── Rewrite / Regenerate ────────────────────────────────────────────────────────

  Endpoint: POST /api/transcribe/rewrite
  First rewrite: Structure-preserving rewrite of original transcript
  Regenerate: Variant of previous rewrite (same talk track, different words)

  Validation (lib/regenerate-validation.ts):
    - Section count must match ±1
    - CTA keyword overlap ≥ 50%
    - Word count within ±15%
    - Auto-retry with strict prompt on failure

  Variation: POST /api/transcribe/variation
  UI: components/TranscriberCore.tsx

── Persona-to-Avatar Mapping ───────────────────────────────────────────────────

  lib/heygen-personas.ts — maps transcriber personas to HeyGen avatar + voice
  Used when rendering the rewritten script as avatar video.

Mission Control (mc.flashflowai.com) is an external operations dashboard.
FlashFlow posts documents, tasks, and status updates to it.

── HTTP Client ─────────────────────────────────────────────────────────────────

  lib/flashflow/mission-control.ts
  Base URL: MC_BASE_URL || 'https://mc.flashflowai.com'

  Token resolution:
    1. MISSION_CONTROL_TOKEN (preferred)
    2. MISSION_CONTROL_AGENT_TOKEN (agent-only fallback)

  Every request sends BOTH headers:
    Authorization: Bearer <token>
    x-service-token: <token>

  Self-healing: On 401, calls /api/auth-check for diagnosis, retries once,
  sends Telegram alert if still failing.

  Auth Provider:    Supabase Auth (SSR via @supabase/ssr)
  Session:          Cookie-based (httpOnly, secure)
  RLS:              Row Level Security on all user-facing tables
  Service Role:     SUPABASE_SERVICE_ROLE_KEY for admin/cron operations

  API Auth patterns:
    - User session (cookie-based) for app routes
    - CRON_SECRET Bearer token for cron routes
    - MISSION_CONTROL_TOKEN for MC bridge
    - BROWSER_SERVICE_KEY for HP Windows browser service
    - Stripe webhook signature verification

  Cron auth:
    vercel.json crons send CRON_SECRET in Authorization header
    Routes verify: Authorization === `Bearer ${CRON_SECRET}`

    Session-based with admin role check
    95 admin sub-pages behind auth gate
